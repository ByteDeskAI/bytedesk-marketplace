# Request Reply Workflow

## Intent
Coordinate asynchronous work that still needs a response, without falling back to tight synchronous coupling.

## When To Use
- The caller can wait asynchronously for a bounded response.
- The responder needs independent scaling or reliability policy.
- Correlation and timeout behavior can be explicit.

## Avoid When
- The interaction is a simple local function call.
- The response is not required by the initiating workflow.
- Timeout, retry, and reply ownership are unclear.

## Pattern Set
- Request-Reply for the conversation shape.
- Command Message for the request intent.
- Return Address and Correlation Identifier for reply routing.
- Message Expiration for bounded usefulness.

## Implementation Steps
1. Define the command and reply contracts separately.
2. Add a correlation identifier generated by the requester.
3. Define timeout, expiration, retry, and terminal failure behavior.
4. Keep reply handlers idempotent when retries are possible.

## Verification
- Timed-out requests stop waiting and surface a clear status.
- Late replies are safely ignored or reconciled.
- Logs and traces connect request, handler, and reply.
